## Key Insights

### SmartCommerceAI Phase 1 - 對話系統與資料持久化

這次把 SmartCommerceAI 的第一階段給搞定了，核心功能圍繞在「對話歷史記錄」這塊。

**對話歷史記錄系統**：
- 整合了完整的對話上下文管理，讓 AI 可以記住之前的對話內容
- 這對電商場景很重要，因為客戶可能會分多次詢問同一個商品或訂單
- 實作時考慮了記憶體效率，避免無限制累積對話造成效能問題

**Azure Table Storage**：
- 選擇 Azure Table Storage 做持久化儲存，主要是看上它的 NoSQL 特性和成本效益
- 對話記錄用 PartitionKey (userId) + RowKey (timestamp) 的設計，查詢效率不錯
- 整合過程中遇到連線管理的問題，最後用連線池解決了

**防濫用機制**：
- 加入了 rate limiting 防止惡意濫用
- 設定了合理的對話長度限制和請求頻率控制
- 這層保護很重要，不然 API 成本可能會爆掉

### Tailscale 遠端存取

設定 Tailscale 主要是為了能安全地遠端存取家裡的開發環境。

- 安裝和配置過程蠻順利的，mesh VPN 的概念確實比傳統 VPN 彈性
- 現在可以從外面安全地 SSH 回家裡的機器，不用擔心暴露公網 IP
- 對開發來說方便很多，在外面也能接續工作

### Learning-Diary 倉庫建立

建立了這個學習日誌系統來記錄每天的學習進度和技術心得。

- 用 GitHub repo 管理，搭配自動化 workflow
- 結構簡單清楚：daily-notes/ 放每日記錄，templates/ 放模板
- 目的是養成每天反思和記錄的習慣，也方便以後回顧

## Challenges and Future Improvements

### 當前挑戰

1. **SmartCommerceAI 的對話記憶優化**
   - 目前對話歷史是簡單的線性累積，長時間對話可能影響回應速度
   - Token 使用量需要更好的控制機制

2. **Learning-Diary 的持續性**
   - 建立系統容易，難的是每天持續記錄
   - 需要找到適合的記錄節奏，不要變成負擔

### 改進計畫

1. **SmartCommerceAI 優化方向**
   - 實作對話摘要功能，壓縮舊對話記錄但保留重要資訊
   - 加入向量資料庫（如 Pinecone 或 Qdrant）做語意搜尋
   - 優化 Azure Table Storage 的查詢策略，可能需要加索引或改資料結構
   - 監控和告警系統，追蹤 API 使用量和錯誤率

2. **Tailscale 應用擴展**
   - 研究 Tailscale 的更多功能，像是 subnet routing
   - 評估是否能整合到其他專案的部署流程

3. **Learning-Diary 習慣養成**
   - 設定每天固定時間寫日誌（睡前或早上）
   - 開發簡單的 CLI 工具快速建立和編輯當天記錄
   - 每週做一次回顧，看看學了什麼、哪些可以深入

### 反思

今天完成的這三件事其實都圍繞在「系統化」這個主題：

- SmartCommerceAI 把對話系統化，讓 AI 有記憶
- Tailscale 把遠端存取系統化，讓工作環境無縫接軌
- Learning-Diary 把學習系統化，讓知識累積有跡可循

下一步就是把這些系統都用好、用滿，把「建立工具」的動力轉換成「使用工具」的習慣。

技術上，SmartCommerceAI 的第二階段可以考慮加入更智慧的推薦系統，或是整合更多電商 API。但更重要的是先把 Phase 1 穩定運行，收集真實使用數據，再來決定優化方向。
